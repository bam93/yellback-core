## Summary

Session 3+7 combo landed the detector architecture: a shared `Detector` protocol, `MicDetector` conforming to it, a working `AccelerometerDetector` using IOKit HID to read the Apple SPU accelerometer on Apple Silicon MacBooks, and a `--listen` CLI mode that starts all enabled detectors and streams triggers to stderr. Cooldown was moved OFF detectors (engine-level; Session 5) — detectors now emit at their natural cadence and the engine will filter rapid-fire events before audio playback.

`MicDetector` owns its own `AVAudioEngine` internally, matching the per-detector-independence principle in `ARCHITECTURE.md`. `AccelerometerDetector` matches on IOKit usage page `0xFF00` / usage `0x03` via a clean-room port of the reverse-engineered HID report format (reference: `olvvier/apple-silicon-accelerometer`, MIT). Q16.16 fixed-point parsing is isolated in a pure `parseReport(...)` function so tests feed synthetic 22-byte buffers without IOKit involvement. Reading the sensor at runtime requires root (`sudo yellback --listen`) — `IOHIDManagerOpen` returns `kIOReturnNotPrivileged` otherwise. This is surfaced as a new `DetectorError.needsPrivilegedAccess` case.

Session 3+7 also took the `MicDetector` `onTrigger`/`onIntensity` callbacks and renamed them to match the protocol's `onTriggerEvent`/`onIntensitySignal`, dropping the `Trigger` parameter from `onIntensity` (each detector emits only one trigger type; the tuple was redundant). `isEnabled` is a new runtime gate. Validation-bug caught during the refactor: `testVoiceBandFilterDisabledAcceptsOutOfBandEnergy` previously hard-asserted 1 trigger under the assumption of detector cooldown; now 50Hz at sustained amplitude fires ~3 times per second at sustain cadence, so the assertion became "at least 1."

Session 2 and earlier detail preserved below.

Session 2 implemented the config layer. `EngineConfig` is a tree of typed, Equatable structs (`TriggersConfig`/`ScreamConfig`/`RageTypeConfig`/`DeskBangConfig`/`PrimingConfig`/`AudioConfig`/`LoggingConfig` + `LogLevel`), each with defaults matching `config.example.yaml`. Validation lives on each leaf struct's throwing init (not in `ConfigLoader`), so programmatic construction by the paid Mac app goes through the same validation as YAML loading. `ConfigLoader` is a pure YAML→struct translator that catches `ConfigError.invalidValue` from struct inits and enriches the error with the full dotted field path and the Yams line number before re-throwing. `ConfigLoader.load(from:)` / `.loadFromString(_:)` return a `LoadResult(config, warnings: [ConfigWarning])`. Every validation rule from `CONFIG_SCHEMA.md` lines 137-147 is enforced; malformed YAML carries line numbers where Yams provides them. The CLI accepts `--config <path>` and prints the parsed config. (Append new decisions here as sessions progress. Each entry: date, decision, rationale, session that made it.)

- **[initial]** Two-repo architecture: this repo (public, MIT) + `yellback-mac` (private). Rationale: OSS contributors see clean MIT code; paid app code stays private. See conversation brief.
- **[initial]** No UI framework imports in the core. Rationale: keeps future Rust/Tauri port as translation rather than redesign.
- **[initial]** Detectors emit both discrete events AND continuous intensity signals. Rationale: v1 audio engine uses events; v2 fusion module will use signals. Both live from v1 onward.
- **[initial]** Priming state lives on the engine, not on individual detectors. Rationale: cross-trigger behaviour is engine-level state.
- **[initial]** `.caf` audio format for bundled clips. Rationale: lowest decode latency of formats AVAudioEngine supports.
- **[session 1 / 2026-04-24]** `Package.swift` uses `swift-tools-version:5.9` rather than 5.10 or 6.0. Rationale: 5.9 is the first Swift toolchain shipped with Xcode 15, which is the first Xcode available on macOS 14 (our minimum). This maximises the range of macOS 14+ users who can build the package without upgrading Xcode.
- **[session 1 / 2026-04-24]** Executable product name is `yellback`, target dir/name is `yellback-cli`. Rationale: users run `yellback` at the shell; the source dir name reflects what kind of code it is. Decoupling product name from target name is the standard SwiftPM pattern for this.
- **[session 1 / 2026-04-24]** `Trigger` and `TriggerEvent` live under `Sources/YellBackCore/Signals/` alongside `IntensitySignal` and `PrimingState`. CLAUDE.md's "Code Organization" only explicitly lists `IntensitySignal` and `PrimingState` in `Signals/` but keeping all event- and signal-shaped types together is the natural grouping; the CLAUDE.md comment was descriptive, not exhaustive.
- **[session 1 / 2026-04-24]** `Package.resolved` is committed. Rationale: this is the standard SwiftPM guidance for any package that ships an executable product (locks Yams version for reproducible builds of the CLI). If the lockfile ever causes contributor friction we can revisit.
- **[session 2 / 2026-04-24]** Every config field has a documented default sourced from `config.example.yaml`. Missing individual fields inside a present block are silently filled with defaults (not rejected). Rationale: consistent with the "whole detector block omittable → enabled:false" semantics already in the schema, and friendlier to users hand-editing partial configs. The canonical source of truth for default values is the `public init(...)` parameter defaults on each nested config struct.
- **[session 2 / 2026-04-24]** Unknown keys never fail — they surface as `ConfigWarning.unknownKey`, including nested paths (e.g. `triggers.scream.unknown_tweak`). Rationale: forward-compat. Older binaries running a newer user's config should degrade gracefully rather than refuse to start. The schema's original "unknown top-level keys warn" rule is applied recursively.
- **[session 2 / 2026-04-24]** `ConfigLoader` does NOT hit the filesystem to validate `packs_directory` at load time (though the schema rule says unreadable/uncreatable paths should fail loading). Rationale: keeps `ConfigLoader` hermetic and its tests pure. The check is deferred to `YellBackEngine.start()`, which is where we first actually need to read the directory. The rule will still be enforced — just later.
- **[session 2 / 2026-04-24]** Warnings are returned in `LoadResult.warnings` rather than `print`ed from inside the loader. Rationale: testability (XCTest can assert on the warning list) and separation of concerns (CLI decides stderr formatting; paid app decides UI rendering).
- **[session 2 / 2026-04-24]** `ConfigLoader` walks the Yams `Node` tree manually rather than using `YAMLDecoder` + `Codable`. Rationale: gives us per-node `mark.line` for precise validation error locations, and lets us distinguish "field missing" from "field null" without surrogate sentinel types. The cost is ~300 lines of hand-written traversal, but the tree is shallow and static, and the parsers are trivial once primitives are factored out.
- **[session 2 / 2026-04-24]** `yellback-cli` uses hand-rolled `CommandLine.arguments` parsing rather than adopting `swift-argument-parser`. Rationale: the current CLI surface is just `--config <path>` + `--help`; a dependency for that would be overkill. Revisit when we add a second real flag (e.g. `--pack`, `--self-test`).
- **[session 2 / 2026-04-24]** All config types are `Equatable`. Rationale: makes test assertions trivial (`XCTAssertEqual(config, EngineConfig())` etc.), and is a reasonable promise to callers who want to diff configs in the paid app's settings UI. `PermissionStatus` is still NOT `Equatable` — belongs to a different session when we wire permissions.
- **[session 2 addendum / 2026-04-24]** Validation rules live on the throwing inits of each leaf config struct, NOT in `ConfigLoader`. Rationale: `EngineConfig` and its nested types are part of `yellback-core`'s public API — the paid Mac app constructs them directly from its settings UI rather than round-tripping through YAML. Keeping validation in `ConfigLoader` meant programmatic construction silently accepted invalid state (e.g. `ScreamConfig(dbfsThreshold: 999)`). Moving validation onto struct inits means both paths — file load and programmatic construction — are validated through the same code. `ConfigLoader` becomes a pure translator that catches struct-init errors and enriches them with the YAML-specific context (full dotted field path + line number from the originating Yams node). - **[session 2 addendum / 2026-04-24]** Only leaf config structs with actual rules have throwing inits (`ScreamConfig`, `RageTypeConfig`, `DeskBangConfig`, `PrimingConfig`, `AudioConfig`). Aggregator types (`TriggersConfig`, `LoggingConfig`, `EngineConfig` itself) don't throw — if every leaf is valid, the aggregate is valid. Rationale: avoids `throws` ceremony on types that have no rules. When we later add cross-field invariants (e.g. "if scream.enabled, audio.pack must resolve to a real pack"), the relevant aggregator grows a throwing init then.
- **[session 2 addendum / 2026-04-24]** Struct inits throw with snake_case field names (`dbfs_threshold`, not `dbfsThreshold`), matching `CONFIG_SCHEMA.md` and the YAML users edit. Rationale: error messages read identically whether the error originated from YAML loading or programmatic construction, so a user debugging from a log line doesn't need to mentally map camelCase back to the docs. The cost is a small amount of duplication between Swift property names and error-reporting names.
- **[session 2 addendum / 2026-04-24]** `.default` accessors on throwing leaf structs use `try!` against the known-valid default values (e.g. `public static let default = try! ScreamConfig()`). Rationale: `try!` gives a loud, immediate crash at first module import if anyone accidentally changes a default value to something that violates a rule — a failure mode that's far preferable to silently shipping invalid defaults.

- **[session 3 / 2026-04-25]** `MicDetector` is buffer-in (not tap-owning) as its primary interface. The scope originally said "MicDetector owns an AVAudioEngine input tap" — that was refined at session start to a split design: `process(buffer:)` is the testable core (called from either the tap or a synthetic test buffer), and `start(on: AVAudioInputNode)` is a thin convenience wrapper that installs a tap and forwards to `process()`. Rationale: tests never need an AVAudioEngine or mic permission; they feed synthetic buffers directly. Live code gets a one-line "start listening" API.
- **[session 3 / 2026-04-25]** Sustain and cooldown logic use a sample-accurate monotonic clock derived from `processedSamples / sampleRate`, NOT wall-clock `Date()` and NOT `AVAudioTime`. Rationale: synthetic test buffers can advance "time" deterministically without real wall-clock passage, so a test feeding 350ms of sine samples genuinely exercises the 300ms sustain boundary. Later detectors (KeyboardDetector, AccelerometerDetector) should follow the same pattern for their respective privacy contracts.
- **[session 3 / 2026-04-25]** Band-pass filter is a hand-rolled cascade of two RBJ-cookbook Butterworth biquads (HPF @ 200Hz + LPF @ 3kHz), not `AVAudioUnitEQ` and not `vDSP_biquad`. Rationale: AVAudioUnitEQ would force `MicDetector` into an AVAudioEngine graph for filtering, which breaks the clean buffer-in API and makes tests need an engine. vDSP is a drop-in optimisation if profiling ever shows we need it (today 16 tests processing thousands of synthetic buffers complete in 0.7s — we don't). The hand-rolled biquad is ~40 lines, zero-dependency, and trivially testable.
- **[session 3 / 2026-04-25]** `AudioFixtures` generates synthetic PCM buffers at test time rather than committing real recordings. Rationale: real recordings are (a) binary blobs in git, (b) only vaguely documented in terms of their acoustic properties, and (c) hard to regenerate if lost. Synthesised sine/silence/noise lets tests say "a 1kHz tone at amplitude 0.5 for 350ms" and assert the exact RMS (`0.5/√2 ≈ 0.354`) and dBFS (`≈ -9`) that will result. Later detector sessions can extend `AudioFixtures` for motion-sensor traces and keystroke streams using the same pattern.
- **[session 3 addendum / 2026-04-25]** Every detector exposes an engine-settable `primingMultiplier: Double` property (default `1.0`). `YellBackEngine` (Session 5) sets this to the configured priming multiplier when its owned `PrimingState` transitions. Detectors apply the multiplier to their effective threshold at trigger-evaluation time. Rationale: ARCHITECTURE.md line 33 says "detectors consult [priming state] before firing — they ask 'given my current reading and the priming state, should I fire?' rather than applying thresholds independently." The initial Session 3 landing missed this hook; the detector had an immutable `ScreamConfig` with no injection point for the engine. Landing the hook in Session 3 means Session 5 is pure wiring rather than MicDetector rework. The multiplier is applied in RMS space per `CONFIG_SCHEMA.md` line 86 (`effective_rms = base_rms × mult`), which translates to a dBFS offset of `20·log10(mult)` in the threshold check.
- **[session 3 addendum / 2026-04-25]** `TriggerEvent.wasPrimed` is computed precisely: `true` iff the firing only happened because of priming — i.e. `dbfs` was below the base threshold but above the primed effective threshold. If the signal was loud enough to fire without priming, `wasPrimed` is `false` even when `primingMultiplier < 1.0`. Rationale: the public API's docstring on `wasPrimed` says "Did priming state affect this firing?", and the only honest answer is to check whether priming was the proximate cause. A looser interpretation (`wasPrimed = (primingMultiplier != 1.0)`) would flag every firing during an active priming window, even ones that would have happened anyway — misleading for consumers trying to understand trigger causation.
- **[session 3+7 combo / 2026-04-25]** Every detector conforms to a shared public `Detector` protocol (`Sources/YellBackCore/Detectors/Detector.swift`). The protocol enforces the signal/event duality at the type level — a detector can't "forget" to expose an intensity-signal callback. It also bans cooldown logic from the detector surface: `start()` / `stop()` are the only lifecycle methods, and there's no detector-level cooldown parameter or state. This sets the pattern for Sessions 5/6 (engine wiring and KeyboardDetector).
- **[session 3+7 combo / 2026-04-25]** Cooldown enforcement moved off detectors entirely. Detectors emit at their natural cadence (sustain cadence for mic, per-sample for accelerometer); the engine will filter rapid-fire events before playback in Session 5. Rationale: priming state lives on the engine and may interact with cooldown timing; the engine is the natural owner of cross-cutting policy; the engine needs to see every raw event for stats (`SessionStats` uses all emissions, not just the audio-triggering ones); and detectors stay testable as pure transformers. The `ScreamConfig.cooldownSeconds` and `DeskBangConfig.cooldownSeconds` fields are still in the config schema — they're now engine inputs, not detector inputs. Engine wiring (Session 5) will read them.
- **[session 3+7 combo / 2026-04-25]** `MicDetector` owns its own `AVAudioEngine` instance. `SoundEngine` (Session 4) will own a separate engine for output. Apple's docs are fine with multiple engines per process, and this matches the per-detector-independence principle. If profiling ever shows the dual-engine cost is material, the engines can be unified in Session 5 when the audio pipeline is wired end-to-end.
- **[session 3+7 combo / 2026-04-25]** `AccelerometerDetector` uses IOKit HID reading `AppleSPUHIDDevice`, NOT CoreMotion. CoreMotion's motion APIs (`CMMotionManager`, `CMSensorRecorder`, `CMMotionActivityManager`) are `API_UNAVAILABLE(macos)` — confirmed via SDK headers. The project's original assumption that CoreMotion would work for desk-bang detection on macOS was wrong. The IOKit approach is a clean-room port of the reverse-engineered HID report format documented in `olvvier/apple-silicon-accelerometer` (MIT-licensed). This is an undocumented API — Apple could break it in any macOS release. Currently works through macOS 15.x on M1 Pro/Max/Ultra + all M2/M3/M4 Macs. Does not work on base M1 laptops or Mac desktops (Mac mini, Mac Studio, Mac Pro) — they have no sensor.
- **[session 3+7 combo / 2026-04-25]** `AccelerometerDetector.start()` requires root. `IOHIDManagerOpen` returns `kIOReturnNotPrivileged` without it, and there's no entitlement that grants this access. Surfaced as `DetectorError.needsPrivilegedAccess`. The CLI documents `sudo yellback --listen` in its usage text; the paid Mac app will ship a privileged helper via `SMAppService.daemon`. Not a blocker for Session 5 since the engine-level code runs as whatever process it's embedded in.
- **[session 3+7 combo / 2026-04-25]** Binary HID report parsing lives in a pure `AccelerometerDetector.parseReport(_:length:)` static function, decoupled from IOKit. Tests feed synthetic 22-byte buffers. Rationale: the IOKit integration can't run in CI (needs root and a physical sensor), but the bit-twiddling is where bugs hide. `loadUnaligned(fromByteOffset:as:)` is required — offsets 6/10/14 aren't 4-byte-aligned and `.load(fromByteOffset:as:)` traps on unaligned reads.
- **[session 3+7 combo / 2026-04-25]** Detectors are `public`, not internal. Rationale: the CLI's `--listen` mode instantiates them directly, and the paid Mac app will too (at least in v1 before the engine is fully baked). Revisit when Session 5 has `YellBackEngine` as the primary consumer — we may narrow visibility then, or keep detectors as a public API so advanced consumers can build their own engine.
